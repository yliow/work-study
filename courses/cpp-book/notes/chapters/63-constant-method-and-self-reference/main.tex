You can (and should) think of the ``stuff'' on the green band (the

not in this class (i.e. methods of another class).

When an object \texttt{obj} invokes a method \texttt{m}, a special pointer
\texttt{this} is passed into \texttt{m} automatically. The important thing
to remember is that:

\EMPHASIZE{\texttt{this} points to \texttt{obj}}

If an object \texttt{obj} has member variable \texttt{x\_} and \texttt{this}
is a pointer to \texttt{obj}, then the following are the same:

\texttt{obj.x\_} \hspace{1cm} \texttt{(*this).x\_} \hspace{1cm} \texttt{this->x\_}

This is the same notation used for pointers to \texttt{struct} variables.
You should \EMPHASIZE{\texttt{not}} use \texttt{(*this).x\_}. Write
\texttt{this->x\_} instead.

(There is an exception. Static methods do not have the \texttt{this}
secret parameter discussed here. We will talk about static methods
later.)

\EMPHASIZE{WARNING:} \texttt{*p.x} means \texttt{*(p.x)} because \texttt{.} has
higher precedence than \texttt{*}. Make sure you try changing the

\newpage\EMPHASIZE{When do you use this?}

\EMPHASIZE{Standard Practice:} Don't use \texttt{this} in a method just to access members or member functions. In other words, in a method

\begin{itemize}
\item \texttt{this->x} should be \texttt{x}
\item \texttt{this->m()} should be \texttt{m()}
\end{itemize}

For instance, this does work:

\begin{consolethree}[escapeinside=||]
...
class Date
{
public:
    ...
    int get_year()
    {
        return |\EMPHASIZE{this->yyyy\_;}|
    }
    void set_year(int y)
    {
        |\EMPHASIZE{this->yyyy\_}| = y;
    }
    ...
    void add_m_y(int m, int d)
    {
        |\EMPHASIZE{this->add\_m(m);}|
        |\EMPHASIZE{this->add\_d(d);}|
    }
private:
    ...
};
\end{consolethree}

but \EMPHASIZE{DON'T DO THAT!!!} Do this:

\begin{consolethree}[escapeinside=||]
...
class Date
{
public:
    ...
    int get_year()
    {
        return |\EMPHASIZE{yyyy\_;}|
    }
    void set_year(int y)
    {
        |\EMPHASIZE{yyyy\_}| = y;
    }
    ...
    void add_m_y(int m, int d)
    {
        |\EMPHASIZE{add\_m(m);}|
        |\EMPHASIZE{add\_d(d);}|
    }
private:
    ...
};
\end{consolethree}

The pointer \texttt{this} is used when it's used on its own either as
\texttt{this} or \texttt{*this}. In particular \texttt{*this} can be used for
returning a copy of or a reference to the object making the method
invocation and \texttt{this} can be used for address comparison
(you'll see this later when we need to compare address
values when we overload the assignment operator \texttt{operator=}.)

(There are exceptions ... later. But the above two uses are good
enough.)


\newpage\EMPHASIZE{Returning a copy of *this}

\begin{consolethree}[escapeinside=||]
...
class Int
{
    ...
    Int max(const Int & c)
    {
        if (c.x_ > x_)
            return c;
        else
            return |\EMPHASIZE{*this;}|
    }
    ...
private:
    int x_;
};
...
\end{consolethree}

And now you can try this:

\begin{consolethree}[escapeinside=||]
#include "Int.h"

int main()
{
    Int a(100), b(1);
    Int c = a.max(b); // a.max(b) should return an
                      // Int object with x_ value of
                      // 100
    c.print();
    return 0;
}
\end{consolethree}

Of course, this is better:

\begin{consolethree}[escapeinside=||]
...
class Int
{
public:
    ...
    Int max(const Int & c)
    {
        return (c.x_ > x_ ? c : *this);
    }
    ...
private:
    int x_;
};
...
\end{consolethree}

Get it? In this case, we are returning a copy of the \texttt{Int} object
that \texttt{this} points to, i.e., the object invoking the method.

Notice however that you can also do this:

\begin{consolethree}[escapeinside=||]
...
class Int
{
    ...
    Int & max(const Int |\EMPHASIZE{\&}| c)
    {
        return (c.x_ > x_ ? c : *this);
    }
    ...
private:
    int x_;
};
...
\end{consolethree}

In this case, a \EMPHASIZE{reference is returned}.

Why is this important? Look at this code:

\begin{consolethree}[escapeinside=||]
#include "Int.h"

int main()
{
    Int a(100), b(1);
    Int c = a.max(b);
    c.print();
    return 0;
}
\end{consolethree}

This means if the return of \texttt{Int::max()} is an \texttt{Int} and not
an \texttt{Int \&}

\begin{consolethree}[escapeinside=||]
...
class Int
{
public:
    ...
    |\EMPHASIZE{Int}| max(const Int & c)
    {
        ...
    }
    ...
};
...
\end{consolethree}

then an \texttt{Int} value is returned (it has the same values as \texttt{a}
or \texttt{b}). This \texttt{Int} value is used for the invocation of the
\texttt{Int} copy constructor to initialize \texttt{c}.

Make sure you read the last two sentences again (and again and again).

However if a reference is returned:

\begin{consolethree}[escapeinside=||]
...
class Int
{
public:
    ...
    |\EMPHASIZE{Int \&}| max(const Int & c)
    {
        ...
    }
    ...
};
...
\end{consolethree}

then back in \texttt{main()},
\begin{consolethree}
#include "Int.h"

int main()
{
    Int a(100), b(1);
    Int c = a.max(b);
    c.print();
    
    return 0;
}
\end{consolethree}

on returning from \texttt{Int::max()}, a reference to a or to b is used by
the copy constructor to initialize \texttt{c}. There's no
extra \texttt{Int} object created by \texttt{Int::max()} that is used to
initialize \texttt{c}.

Let me repeat again: when you return \texttt{Int} (not \texttt{Int \&}), a
copy of the object to be returned is made (by using the copy
cconstructor).
\begin{consolethree}[escapeinside=||]
...
class Int
{
public:
    ...
    |\EMPHASIZE{Int}| max(const Int & c)
    {|\xsidebox[0cm]{0cm}{}{ctorbx}{Copy constructor is called using \texttt{c} or \texttt{*this}!!! Because return type is \texttt{Int} (and not \texttt{Int \&}) Remember that!!!}|
        return (c.x > x ? c : *this);
    }
    ...
private:
\end{consolethree}
Whereas if you return \texttt{Int \&}, no \texttt{Int} object is created.
The copy constructor used to create \texttt{c} will simply reference
\texttt{a} or \texttt{b}.


\newpage\EMPHASIZE{Returning references}

In the previous section, I have shown you an example of returning a
reference.

Certain methods and operators update the values of the variables or
objects and return the values of the variables or objects. Run this:

\begin{consolethree}[escapeinside=||]
int x = 5;
x++;
std::cout << x << std::endl;
int y = x++;
std::cout << x << ',' << y << std::endl;
y = (x += 5);
std::cout << x << ',' << y << std::endl;
(x += 5) += 5;
std::cout << x << std::endl;
\end{consolethree}

Don't overdo this! Why? Because

\begin{consolethree}[escapeinside=||]
y = (x += 5);
\end{consolethree}

does too many things. A function/method/operator should do
\EMPHASIZE{\texttt{one}} thing to avoid confusion. This is clearly easier to
understand:

\begin{consolethree}[escapeinside=||]
x += 5;
y = x;
\end{consolethree}

Note that the \texttt{+=} when called twice:

\begin{consolethree}[escapeinside=||]
(x += 5) += 5;
\end{consolethree}

operates on the same \texttt{x}. This means that \texttt{+=}, besides
incrementing \texttt{x}, also returns a reference to x so that the second
\texttt{+=} operators on the same \texttt{x}. If \texttt{+=} returns a new
\texttt{int} value that has the same value as \texttt{x} after incrementing,
then the second \texttt{+=} would have operated on this new \texttt{int}
value and not on \texttt{x}. Therefore the right thing to do is to return
a reference to \texttt{x} and not a copy of \texttt{x}.

Of course returning a reference is faster than returning an object since
objects tend to consume more memory. References are implemented using
pointers and pointers take up very little memory.

Refer to the \texttt{Date} example. Suppose the \texttt{add\_y} method is to
add an integer value to the year value, \texttt{and} return the value of
the object.

\begin{consolethree}[escapeinside=||]
// Date.h
...
class Date
{
public:
    ...
    |\EMPHASIZE{Date \&}| add_y(int);
    ...
private:
    int yyyy_, mm_, dd_;
};
...
\end{consolethree}

\begin{consolethree}[escapeinside=||]
// Date.cpp
...
|\EMPHASIZE{Date \&}| Date::add_y(int inc)
{
    yyyy_ += inc;
    |\EMPHASIZE{return *this;}|
}
...
\end{consolethree}

With the above, I can now do this:

\begin{consolethree}[escapeinside=||]
Date today(1, 1, 1970);
today.add_y(1).add_y(3);
\end{consolethree}

The first invocation of \texttt{add\_y} will add 1 to the year of
\texttt{today} and then a reference to \texttt{today} is returned. Therefore
the second \texttt{add\_y} is called by this reference, which means that
the second \texttt{add\_y} is called by \texttt{today}.

However if you do this:

\begin{consolethree}[escapeinside=||]
// Date.cpp
...
|\EMPHASIZE{Date}| Date::add_y(int inc)
{
    y += inc;
    |\EMPHASIZE{return *this;}|
}
...
\end{consolethree}

i.e., return a \texttt{Date} object instead of a reference, then
\texttt{Date}'s copy constructor is used to create a new
\texttt{Date} object from \texttt{*this}. This object is returned to
\texttt{main()} and the \texttt{add\_y} is called by this object so that
\texttt{today} does not have its year incremented by 3.

We now look at returning references from a function call or method call.
There's nothing new here about references.

Of course you can also return a reference to a member (yes I know

But remember about information hiding. Clients usually should not have
direct access to instance variables. The class (and the objects) should
provide services to clients through methods and hide internal
implementation from outsiders. This makes the software more
maintainable.

And you definitely cannot do this in a method:

\begin{consolethree}[escapeinside=||]

\newpage\EMPHASIZE{Temporary object values}

Most of the time we have objects with names:

\begin{consolethree}[escapeinside=||]
Date date(1970, 1, 1);
date.print();
\end{consolethree}

Of course you can create object values not bound to names:

\begin{consolethree}[escapeinside=||]
Date(1970, 1, 1).print();
\end{consolethree}

(I've already talked about this.)

In the second case above, we have a temporary object, or an object value
that does not belong to an object name.

In the \texttt{Int} class, if you have operator+ returns object values,
you might see temporary objects:

\begin{consolethree}[escapeinside=||]
Int i(5), j(7), k (8);
Int z = i + j + k; // i + j is temporary object
                   // this temporary object is
                   // added with k to produce
                   // another temporary object
\end{consolethree}

Of course, just like a method, a function can return an object value

\begin{consolethree}[escapeinside=||]
Date rand_date()
{
    return Date(rand()%10 + 1,
                rand()%10 + 1, rand());
}
\end{consolethree}

Of course this is different from returning a reference:

\begin{consolethree}[escapeinside=||]
Date & Date::f()
{
    return (*this);
}

int main()
{
    Date date(1, 1, 1970);
    date.f(); // A reference to date
              // is returned
}
\end{consolethree}

A temporary object can be destroyed immediately after the expression
containing the temporary object is computed.

There is an exception ... you can create an object name and bind it
to a temporary object value. In this case the temporary object value is
not destroyed immediately. For instance

\begin{consolethree}[escapeinside=||]
int main()
{
    Date date1 = rand_date();
    ...
}
\end{consolethree}

The value of \texttt{date1} is actually the same as the value of the
object returned by \texttt{rand\_date()}, i.e., there is no copy
constructor call. To verify this add some print statements:

\begin{consolethree}[escapeinside=||]
Date randdate()
{
    Date ret(rand() % 10 + 1,
             rand() % 10 + 1, rand());
    std::cout << &ret << '\n';
    return ret;
}

int main()
{
    Date date1 = f();
    std::cout << &date1 << '\n';
}
\end{consolethree}

You can also insert print statements in the \texttt{Date} destructor to
verify that there is only one destructor call, not two, i.e., in the
above, there is really only one \texttt{Date} value.

\begin{consolethree}[escapeinside=||]
rand_date(); // Date value returned by
             // rand_date() is not bound to a
             // name. It's destroyed.
             // Destructor is called.

Date date1 = rand_date(); // Date value bound to
                          // date1. Destructor not
                          // called yet

\newpage\EMPHASIZE{Forcing a reference parameter to be constant}

Recall: When writing a function, it's a good idea to use \texttt{const}
for arguments that should not be changed. This is the case for both
(regular) functions and methods of a class:

\begin{consolethree}[escapeinside=||]
// function
void f(|\EMPHASIZE{const}| C & c) { ... }

// method in class C
void C::g(|\EMPHASIZE{const}| C & c) { ... }
\end{consolethree}

The following example shows 4 possible ways of pass-by-reference:

\begin{itemize}
\item Non-const to non-const \&
\item Non-const to const \&
\item Const to non-const \&
\item Const to const \&
\end{itemize}

Which ones work and which do not? Here's an experiment
that will help.

\begin{consolethree}[escapeinside=||]
void f(const int & i) {}
void g(int & i) {}

int main()
{
    int i = 0;
    const int j = 0;
    f(i); // nonconst passed to const reference
    f(j); // const passed to const reference
    g(i); // nonconst passed to nonconst reference
    g(j); // const passed to nonconst reference
    return 0;
}
\end{consolethree}

First figure out on your own which is reasonable. Compile this and read
the error message. Correct and check by compiling the program.

Answer:

\begin{itemize}
\item Non-const to non-const \&         OK
\item Non-const to const \&             OK
\item Const to non-const \&             BAD!!!
\item Const to const \&                 OK
\end{itemize}

Why?

\begin{consolethree}[escapeinside=||]
void g(int & i) {} // i not constant
                   // g() can change i

int main()
{
    const int j = 0; // j cannot be changed
    g(j); // oops ... now what?
}
\end{consolethree}

\newpage\EMPHASIZE{Forcing a return reference to be constant}

Recall this example from above:

\begin{consolethree}[escapeinside=||]
class Point2d
{
public:
    int & get_x()
    {
        return x_;
    }
    void print()
    {
        std::cout << x_ << ',' << y_ << '\n';
    }
private:
    int x_, y_;
};

int main()
{
    Point2d p;
    p.print();
    int & a = p.get_x();
    a = 0;
    p.print();
    return 0;
}
\end{consolethree}

The \texttt{p.get\_x()} method returns a reference to \texttt{p.x\_}. This
reference is then given to \texttt{a}:

\begin{consolethree}[escapeinside=||]
int & a = p.get_x();
\end{consolethree}

This means that a is a reference to p.x\_. Therefore changing a will
mean changing p.x\_.

But remember what I said earlier about information hiding. Clients
usually should not have direct access to instance variables. The class
(and the objects) should provide services to clients through methods and
hide internal implementation from outsiders. This makes the software
more maintainable. So to prevent \texttt{main()} from changing
\texttt{p.x\_} directly, you can do this:

\begin{consolethree}[escapeinside=||]
class Point2d
{
public:
    |\EMPHASIZE{const}| int & get_x()
    {
        return x_;
    }
    ...
\end{consolethree}

The reference returned will not be able to change the x\_ of the object.
In fact, now the \texttt{main()} above cannot be compiled. Why?

\begin{consolethree}[escapeinside=||]
class Point2d
{
public:
    const int & get_x()
    {
        return x_;
    }
    ...
};

int main()
{
    Point2d p;
    p.print();
    int & a = p.get_x();
    ...
}
\end{consolethree}

Because the \texttt{a} in \texttt{main()} is a reference that can
potentially attempt to the the value of \texttt{p.x\_}. The issue is not
whether \texttt{a} will actually change or not. The only way for the above
to compile is to make it impossible for a to change, i.e., you would
have to do this:

\begin{consolethree}[escapeinside=||]
class Point2d
{
public:
    const int & get_x()
    {
        return x_;
    }
    ...
};

int main()
{
    Point2d p;
    p.print();
    |\EMPHASIZE{const}| int & a = p.get_x();
    ...
}
\end{consolethree}

\newpage\EMPHASIZE{Summary}

\begin{itemize}
\item When an object \texttt{obj} invokes a method \texttt{m}, a pointer called
  \texttt{this} is in the scope of \texttt{m}. The \texttt{this} pointer
  points to \texttt{obj}. In other words, in a method, the \texttt{this}
  pointer always point to the object invoking the method. (See section
  on static methods for exception.)

\item In the body of a method, if an object \texttt{obj} has instance variable
  \texttt{x\_}, then the following are the same:

\texttt{x\_} is the same as \texttt{this->x\_}

\item In the body of a method, if an object \texttt{obj} has another method
  called \texttt{m}, then the following are the same:

\texttt{m()} is the same as \texttt{this->m()}

\item A method can return a copy of the value of the object of class
  \texttt{C}:

\begin{consolethree}[escapeinside=||]
C C::m()
{
    ...
    return *this;
}
\end{consolethree}

\item or a reference to itself:

\begin{consolethree}[escapeinside=||]
C & C::m()
{
    ...
    return *this;
}
\end{consolethree}

\item If a method returns a reference to itself, then you can chain up
  methods:

\texttt{obj.m().m1()}

  where \texttt{m1} is another method and \texttt{m1} is also invoked by
  \texttt{obj}.

\item WARNING: If \texttt{m} does not return a reference of the invoking
  object, then in the following:

\texttt{obj.m().m1()}

  \texttt{m1()} is not invoked by \texttt{obj}, but rather by a copy of
  \texttt{obj}.

\item A constant method is a method that cannot modify the object invoking
  the method.

\item A method is made constant by putting the word \texttt{const} on the
  right of the method header. This is done in the header file and if the
  method is defined outside the class definition, then it must also be
  done in the definition of the method.
\end{itemize}
